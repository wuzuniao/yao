# 首页打卡逻辑梳理 Spec

## Why

首页（`frontend/src/pages/index/index.vue`）的打卡按钮逻辑涉及多种状态切换、时间窗口划分、提醒间隔判定、任务卡片交互等，规则交织且与时间强相关。本文档以文字 + 一天时间流程图的形式，完整梳理首页打卡按钮与任务卡片在各个场景下的**期望行为**，作为后续代码修改的基准。

> **重要**：本 spec 完全以用户本次给出的逻辑为准。若与目录中已有逻辑存在冲突，一律以本 spec 为准。

## What Changes

- 本 spec 为**纯文档梳理**，不修改任何代码。
- 输出：以文字描述 + 一天时间流程图，完整展示首页打卡按钮状态演化与任务卡片交互逻辑。
- 与旧 spec 的**关键冲突点**（已以新逻辑覆盖）：
  - **开始打卡时间**：旧 `t-60`（前 1 小时）→ 新 `t-120`（前 2 小时）
  - **"立即打卡"窗口**：旧从 `t-60` 持续到打卡为止 → 新从 `t-120` 开始，匹配区间内未打卡即为"立即打卡"
  - **"已打卡"持续时长**：旧到 `next_t-60` 或 24:00 → 新到匹配区间结束（按相邻提醒的中点划分），最后一次到 24:00
  - **打卡记录匹配**：旧 `[t-60, t+60]` 区间 → 新按相邻中点划分的匹配区间判定，**覆盖全天 0:00-24:00 无间隙、无留白**，第一个提醒从 `0:00` 开始，最后一个到 `24:00` 结束
  - **新增防抖规则**：同一任务 3 秒内只允许点击一次
  - **新增不在线不检查**：用户不在线（页面不在前台）时停止定时刷新，`onHide` 清除定时器

## Impact

- 受影响代码（仅作为梳理对象，不改动）：[index.vue](file:///d:/wuzuniao/yao/frontend/src/pages/index/index.vue)
- 受影响能力：首页打卡交互、状态机判定、定时刷新机制、打卡记录匹配判定、任务卡片交互与默认显示顺序。

### Requirement: 三种"无打卡任务"（disabled）场景

系统 SHALL 在以下三种场景下将按钮状态设为 `disabled`（灰色"无打卡任务"）。

#### Scenario: 情况 1 — 用户未登录

- **WHEN** `userStore.userInfo` 为空（`isLoggedIn = false`）
- **THEN** `activePlans` 为空，显示空状态文本"欢迎使用无足鸟按时吃药打卡！"
- **AND** 任务卡片区域隐藏
- **AND** 按钮状态 = `disabled`（灰色"无打卡任务"，不显示图标，不可点击）

#### Scenario: 情况 2 — 已登录但未创建计划

- **WHEN** 已登录，但 `listPlans` 返回的计划中 `status === 1` 的为空（用户未创建计划或全部已停用）
- **THEN** 显示空状态文本"请先到设置界面创建您的打卡计划，常见问题可参考同界面里的帮助中心。"
- **AND** 任务卡片区域隐藏
- **AND** 按钮状态 = `disabled`（灰色"无打卡任务"）

#### Scenario: 情况 3 — 已登录但当前时间不属于进行中计划的打卡日期范围

- **WHEN** 已登录、有进行中计划，但今日日期 `today` 满足 `today < plan.start_date` 或 `today > plan.end_date`
- **THEN** 按钮状态 = `disabled`（灰色"无打卡任务"）

### Requirement: 打卡记录匹配区间划分规则

系统 SHALL 明确定义每个提醒时间的"匹配区间"，用于判定当前时间应归属于哪个提醒、以及该提醒是否已打卡。**匹配区间按相邻提醒的中点划分，覆盖全天 0:00-24:00，无间隙、无留白。**

#### 核心定义

- **开始打卡时间** = 第一个提醒时间 `t_1` 前 2 小时（`t_1.minutes - 120`）
- **第一个提醒时间之前**（`nowMinutes < t_1.minutes - 120`）：按钮为 `waiting`（未到打卡时间）
- **从第一个提醒时间的"开始打卡时间"起**（`nowMinutes >= t_1.minutes - 120`）：进入可打卡时段，按钮为 `active` 或 `done`
- **匹配区间与状态判定的关系**：匹配区间覆盖全天用于查询打卡记录；状态判定（waiting/active/done）在匹配区间内根据当前时间和打卡记录判定

#### Scenario: 第一次提醒时间的匹配区间

- **GIVEN** 第一个提醒时间 `t_1`，与第二个提醒时间 `t_2`
- **THEN** `t_1` 的匹配区间为 `[0:00, midpoint(t_1, t_2)]`
  - 起始 = `0:00`（当日开始，因之前无其他提醒）
  - 结束 = `midpoint(t_1, t_2)`（与第二次提醒时间的中间点，前半部分）
- **AND** 在匹配区间内：
  - 有打卡记录 → `done`（已打卡，从打卡时间持续到匹配区间结束）
  - 无打卡记录 → `active`（若 `nowMinutes >= t_1 - 120`）或 `waiting`（若 `nowMinutes < t_1 - 120`）

#### Scenario: 中间提醒时间的匹配区间

- **GIVEN** 中间提醒时间 `t_i`（`1 < i < n`），与前一个 `t_{i-1}` 和后一个 `t_{i+1}`
- **THEN** `t_i` 的匹配区间为 `[midpoint(t_{i-1}, t_i), midpoint(t_i, t_{i+1})]`
  - 起始 = `midpoint(t_{i-1}, t_i)`（与前一个提醒的中间点，后半部分）
  - 结束 = `midpoint(t_i, t_{i+1})`（与后一个提醒的中间点，前半部分）
- **AND** 在匹配区间内：有打卡记录 → `done`；无打卡记录 → `active`

#### Scenario: 最后一次提醒时间的匹配区间

- **GIVEN** 最后一个提醒时间 `t_n`
- **THEN** `t_n` 的匹配区间为 `[midpoint(t_{n-1}, t_n), 24:00]`
  - 起始 = `midpoint(t_{n-1}, t_n)`（与前一个提醒的中间点，后半部分）
  - 结束 = `24:00`（当日结束，因之后无其他提醒）
- **AND** 在匹配区间内：有打卡记录 → `done`；无打卡记录 → `active`

> 含义：
> - **全天无留白**：从 0:00 到 24:00 被各提醒的匹配区间完全覆盖，任意时刻的打卡记录都能归属到某个提醒。
> - 第一个提醒的匹配区间从 0 点开始（因无更早的提醒），最后一个提醒的匹配区间到 24 点结束（因无更晚的提醒）。
> - 相邻提醒以中点为分界：中点之前属于前一个提醒的匹配区间，中点之后属于后一个提醒的匹配区间。
> - 无论间隔 > 4 小时还是 ≤ 4 小时，匹配区间一律按中点划分，确保全天覆盖。

### Requirement: 打卡记录匹配判定

系统 SHALL 明确说明如何判断"某匹配区间是否已打卡"，用于驱动 `active` / `done` 状态切换。

#### Scenario: 匹配区间内的打卡记录匹配规则

- **GIVEN** 当前时间处于提醒 `t_i` 的匹配区间内
- **WHEN** 判断该匹配区间是否已打卡
- **THEN** 在数据库中查询该匹配区间内是否有打卡记录：
  - 第一次提醒：查询 `[0:00, midpoint(t_1, t_2)]` 区间
  - 中间提醒：查询 `[midpoint(t_{i-1}, t_i), midpoint(t_i, t_{i+1})]` 区间
  - 最后一次提醒：查询 `[midpoint(t_{n-1}, t_n), 24:00]` 区间
- **AND** 有打卡记录 → `done`（已打卡，从打卡时间持续到匹配区间结束）
- **AND** 无打卡记录 → `active`（立即打卡，若已过开始打卡时间）或 `waiting`（若未到开始打卡时间）

> ⚠️ 与旧 spec 差异：旧 spec 用 `[t-60, t+60]` 区间匹配；新逻辑按"中点划分的匹配区间"判定，覆盖全天无留白，第一个提醒从 0 点开始，最后一个到 24 点结束。

### Requirement: 一天时间流程图 — 打卡状态演化

系统 SHALL 以时间流程图展示一天 24 小时内打卡按钮状态的变化，覆盖单提醒、双提醒、多提醒场景。

#### Scenario: 示例 1 — 单提醒时间（12 点）

- **GIVEN** 一天 24 小时，只有中午 12 点一个提醒时间
- **AND** 开始打卡时间 = 12:00 - 2h = 10:00
- **AND** 管辖区间 = \[10:00, 24:00]

```
时间轴    0点 ───────── 10点 ────── 12点 ────────────── 24点
          │            │           │                  │
未打卡    └─ 未到打卡时间 ─┴─ 立即打卡 ─────────────────┘
                        ↑
                   开始打卡时间

7点打卡:  └─ 未到打卡时间 ─┘ ↑长按重置→ 已打卡(7点) ──────── 24点
                           (7点时为waiting，长按3秒重置为active后打卡)

11点打卡: └─ 未到打卡时间 ────┴─ 立即打卡 → 已打卡(11点) ── 24点

16点打卡: └─ 未到打卡时间 ────┴── 立即打卡 ────→ 已打卡(16点) → 24点
```

**规则说明：**

- 每分钟检查：当数据库中当天有一个打卡记录时，按钮显示为"已打卡"状态
- 用户整天都未打卡时：0 点-10 点为"未到打卡时间"；10 点-24 点为"立即打卡"
- 用户在 7 点打卡时：7 点默认为"未到打卡时间"（因 7 < 10），长按 3 秒重置为"立即打卡"后打卡，打卡后"已打卡"持续到 24 点
- 用户在 11 点 / 16 点打卡时：点击打卡后生成记录写入数据库，"已打卡"持续到 24 点

***

#### Scenario: 示例 2 — 双提醒时间（8 点、20 点）

- **GIVEN** 一天 24 小时，有上午 8 点和下午 20 点两个提醒时间
- **AND** 开始打卡时间 = 8 - 2h = 6:00
- **AND** 8 点与 20 点的中点 = 14:00
- **AND** 8 点的匹配区间 = [0:00, 14:00]（从 0 点到中点的前半部分）
- **AND** 20 点的匹配区间 = [14:00, 24:00]（从中点的后半部分到 24 点）

```
时间轴    0点 ── 6点 ─ 8点 ─── 14点(中点) ── 20点 ─── 24点
          │     │     │         │            │         │
          │     │     │         │            │         │
未打卡    └未到  └── 立即打卡(8点匹配区间) ─┴─ 立即打卡(20点匹配区间)─┘
          │     ↑                ↑            ↑
          │  开始打卡时间      8点匹配区间   20点匹配区间
          │  (8点-2h)         结束/20点开始  结束
          │                   (按中点划分)

上午打卡  └未到  └── 立即打卡 → 已打卡(打卡时) ──→ 14点 ┴─ 立即打卡 ─── 24点
                                                  ↑
                                        8点匹配区间结束
                                        20点匹配区间开始

上下午都   └未到  └── 立即打卡 → 已打卡(打卡时) ──→ 14点 ┴─ 立即打卡 → 已打卡 → 24点
打卡                                                 (打卡时)
```

**规则说明：**

- 每分钟检查（用户在线时）：
  - 8 点与 20 点的中点 = 14 点
  - 8 点为第一次提醒，匹配区间 = [0:00, 14:00]（0 点到中点的前半部分）
  - 20 点为最后一次提醒，匹配区间 = [14:00, 24:00]（中点的后半部分到 24 点）
  - 当数据库中 [0:00, 14:00] 有打卡记录时，按钮在 8 点匹配区间内显示"已打卡"
  - 当数据库中 [14:00, 24:00] 有打卡记录时，按钮在 20 点匹配区间内显示"已打卡"
  - 全天无留白，任意时刻打卡记录都能归属到对应提醒
- 用户整天都未打卡时：0 点-6 点为"未到打卡时间"；6 点-24 点为"立即打卡"
- 用户上午打过一次卡时：
  - 0 点-6 点："未到打卡时间"
  - 6 点-14 点：8 点匹配区间，若打过一次卡，从打卡时间开始"已打卡"，持续到 14 点（中点）
  - 14 点-24 点：20 点匹配区间，"立即打卡"
- 用户上午、下午都打卡时：
  - 0 点-6 点："未到打卡时间"
  - 6 点-14 点：8 点匹配区间，打卡后从打卡时间"已打卡"持续到 14 点（中点）
  - 14 点-24 点：20 点匹配区间，打卡后从打卡时间"已打卡"持续到 24 点

#### Scenario: 示例 3 — 多提醒时间通用规则

- **GIVEN** 一天 24 小时，有多个提醒时间
- **THEN** 按相邻中点划分匹配区间，覆盖全天 0:00-24:00，无间隙、无留白

**每分钟检查（用户在线时）：**
- 每分钟从数据库同步打卡记录，根据当前时间所在匹配区间判定状态
- 用户不在线（页面不在前台）时不进行检查

**匹配区间划分规则（统一按中点，不分间隔大小）：**

- 第一次提醒 `t_1` 的匹配区间：`[0:00, midpoint(t_1, t_2)]`（从 0 点到与第二次提醒的中点）
- 中间提醒 `t_i` 的匹配区间：`[midpoint(t_{i-1}, t_i), midpoint(t_i, t_{i+1})]`（从前一个中点到后一个中点）
- 最后一次提醒 `t_n` 的匹配区间：`[midpoint(t_{n-1}, t_n), 24:00]`（从前一个中点到 24 点）
- 用户点击打卡后，从打卡时间开始"已打卡"，持续到匹配区间结束（即下一个中点或 24 点）
- 比第一次提醒时间 2 小时前更早的时间为"未到打卡时间"（`waiting`）
- 匹配区间内无打卡记录且已过开始打卡时间 → `active`（立即打卡）

```
时间轴    0点 ─── t1-2h ── t1 ── 中点(t1,t2) ── t2 ── 中点(t2,t3) ── ... ── 24点
          │        │       │        │            │        │              │
          │  t1匹配区间      │  t2匹配区间         │  tn匹配区间         │
未打卡    └─ 未到 ─┴─ 立即打卡 ─┴── 立即打卡 ─────┴── 立即打卡 ────────┘
                               ↑
                          t1匹配区间结束
                          t2匹配区间开始

t1打卡    └─ 未到 ─┴─ 立即打卡 → 已打卡 ─→ 中点 ┴── 立即打卡 ────────────┘
```

> 关键：无论相邻提醒间隔 > 4 小时还是 ≤ 4 小时，匹配区间一律按中点划分。全天 0:00-24:00 被各提醒的匹配区间完全覆盖，任意时刻的打卡记录都能归属到对应提醒，不存在留白。

### Requirement: 长按 3 秒重置

系统 SHALL 允许在特定状态下长按按钮 3 秒重置为"立即打卡"。

#### Scenario: 可重置的状态

- **WHEN** 按钮处于 `waiting`（未到打卡时间）或 `done`（已打卡）状态
- **AND** 用户长按按钮 3 秒（`handleLongPress` 启动 3000ms 计时器）
- **THEN** `forceActive` 设为 `true`，强制 `checkinState` 返回 `active`
- **AND** 显示"已重置为立即打卡"成功提示
- **AND** 按钮转为红色"立即打卡"，可点击打卡
- **AND** 打卡成功或切换任务后 `forceActive` 重置为 `false`

> 注：`disabled`（无打卡任务）和 `active`（立即打卡）状态不可长按重置。

***

### Requirement: 打卡防抖

系统 SHALL 防止频繁点击生成多条打卡记录。

#### Scenario: 3 秒内只允许一次打卡

- **WHEN** 用户点击"立即打卡"按钮
- **THEN** 同一任务在 3 秒内只允许点击一次
- **AND** 3 秒内的重复点击将被拦截，不触发 `createCheckin` 接口调用

> ⚠️ 新增规则：旧 spec 无此防抖逻辑，需在代码修改时新增。

***

### Requirement: 打卡成功后的状态变化

系统 SHALL 描述用户点击"立即打卡"成功后按钮的状态流转。

#### Scenario: 打卡成功后的即时状态

- **WHEN** 用户在 `active` 状态下点击按钮
- **AND** `createCheckin` 接口返回 `code === 0`
- **THEN** 将该打卡记录纳入本地"匹配打卡记录"列表
- **AND** `forceActive` 重置为 `false`
- **AND** 按钮立即转为 `done` 视觉（绿色"已打卡" + `daka_1.png`），不弹窗提示
- **AND** 单击不再触发打卡

#### Scenario: 打卡成功后 done 状态的持续时长

- **WHEN** 用户完成某提醒时间 `t_i` 的打卡，按钮转为 `done`
- **THEN** `done` 状态持续到下列任一条件：
  1. **匹配区间结束**（即下一个中点 `midpoint(t_i, t_{i+1})`）：进入下一个提醒的匹配区间，按钮重新判定
  2. **当日 24:00 之后**（最后一次提醒的匹配区间结束）：跨日重置为 `waiting`
  3. **切换任务**：按新任务重新判定
  4. **长按 3 秒重置**：强制转为 `active`（用于补打或纠错）

### Requirement: 图标状态自动刷新机制

系统 SHALL 描述按钮状态随时间刷新的机制与边界，包括用户不在线时不进行检查的规则。

#### Scenario: 每分钟定时刷新（用户在线时）

- **GIVEN** 页面 `onShow` 时启动 `refreshTimer`，间隔 `60000ms`（1 分钟）
- **WHEN** 定时器触发
- **THEN** 调用 `loadTodayCheckins` 从数据库同步当前主要卡片计划的今日打卡记录
- **AND** 根据同步后的打卡记录重新判定当前匹配区间是否已打卡，相应设置 `done` 或 `active`
- **AND** 若数据库连接异常，仅打印日志，不阻塞界面，按钮保持默认状态

#### Scenario: 用户不在线时不检查

- **WHEN** 用户不在线（页面不在前台，如切换到其他页面、最小化小程序、锁屏等）
- **THEN** 停止每分钟定时刷新，不进行数据库检查
- **AND** 页面 `onHide` 时清除 `refreshTimer`
- **AND** 用户重新回到页面（`onShow`）时重新启动定时器并立即同步一次打卡记录

#### Scenario: 时间流逝与状态重算的关系（边界说明）

- **GIVEN** `checkinState` 为 `computed`，内部使用 `new Date()` 获取当前时间
- **THEN** `new Date()` 不是 Vue 响应式依赖，时间流逝本身不会自动触发 `checkinState` 重算
- **AND** 状态重算由以下事件触发：
  1. 每分钟定时器更新打卡记录（响应式 ref 变化）
  2. 用户交互（点击次要卡片、选择任务、长按重置、`onShow` 重新加载计划等）
  3. `forceActive` / `primaryPlanId` 等 ref 被修改
- **NOTE** 因此"跨入新匹配区间"或"到达下一次提醒开始打卡时间"的视觉切换，通常在下一次定时器触发或用户交互时才体现，而非严格在边界时刻即时切换。

#### Scenario: 页面显示时刷新

- **WHEN** 页面触发 `onShow`（包括首次进入及从其他页面返回）
- **THEN** 调用 `loadActivePlans` 加载进行中计划
- **AND** 若存在主要卡片计划，调用 `loadTodayCheckins` 同步今日打卡记录
- **AND** 启动每分钟刷新定时器（若尚未启动）

***

### Requirement: 任务卡片交互逻辑

系统 SHALL 描述任务卡片的交互行为规则。

#### Scenario: 打卡只针对主要卡片任务

- **WHEN** 用户点击"立即打卡"按钮
- **THEN** 每次打卡都只生成一次主要卡片中任务的打卡记录
- **AND** 不涉及次要卡片任务

#### Scenario: 点击主要卡片无反应

- **WHEN** 用户点击主要卡片
- **THEN** 无任何反应，不设任何点击事件

#### Scenario: 点击次要卡片互换

- **WHEN** 用户点击次要卡片
- **THEN** 主要卡片中的任务和次要卡片中的任务进行交换
- **AND** 再次点击时再次交换
- **AND** `forceActive` 重置为 `false`
- **AND** 重新调用 `loadTodayCheckins` 加载新主要计划的打卡记录

#### Scenario: 点击次要卡片的"..."展开任务列表

- **WHEN** 用户点击次要卡片右侧的"..."
- **THEN** 展开任务列表弹层
- **AND** 用户点击某项任务后，将选中任务替换到主要卡片
- **AND** 原主要卡片中任务替换为次要
- **AND** `forceActive` 重置为 `false`
- **AND** 重新调用 `loadTodayCheckins`

***

### Requirement: 任务卡片默认显示顺序

系统 SHALL 描述任务卡片的默认显示逻辑与优先级规则。

#### Scenario: 只显示进行中任务

- **WHEN** 加载任务卡片
- **THEN** 只显示进行中任务，即当前时间在任务的打卡日期范围内（`start_date <= today <= end_date`）
- **AND** 按优先级升序排序（priority 值越小优先级越高）

#### Scenario: 只有一个进行中任务

- **WHEN** 进行中任务数量 = 1
- **THEN** 只显示主要卡片，不显示次要卡片，不显示"..."

#### Scenario: 只有两个进行中任务

- **WHEN** 进行中任务数量 = 2
- **THEN** 显示主要卡片和次要卡片，不显示次要卡片的"..."
- **AND** 主要卡片显示规则：
  - 当前时间在提醒时间前后 2 小时范围内的任务显示在主要卡片上
  - 若当前时间都在两个任务的提醒时间范围内，优先级高的为主，显示在主要卡片
  - 当任务优先级一样，先创建的任务为主，显示在主要卡片

#### Scenario: 有多个进行中任务（3 个及以上）

- **WHEN** 进行中任务数量 ≥ 3
- **THEN** 显示主要卡片、次要卡片和次要卡片的"..."
- **AND** 两个卡片的任务默认显示逻辑和两个任务时相同
- **AND** 点击"..."会展开任务列表，可进行任务替换

| 主要卡片显示    | 按 priority 升序取第一个       | 当前时间在提醒时间前后 2 小时范围内的优先；都在范围内时优先级高的为主；优先级相同时先创建的为主 |
