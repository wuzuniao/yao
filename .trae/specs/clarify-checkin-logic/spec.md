# 首页打卡逻辑梳理 Spec

## Why
首页（`frontend/src/pages/index/index.vue`）的打卡按钮逻辑涉及多种状态切换（未登录、无任务、未到时间、可打卡、已完成、刚打卡成功等），状态判定条件交织，且与时间窗口、提醒时间间隔、任务切换、长按重置等交互耦合。本文档以文字形式完整梳理首页打卡按钮的**期望行为**，作为后续优化、排错与代码修改的基准，避免口头理解偏差。

## What Changes
- 本 spec 为**纯文档梳理**，不修改任何代码。
- 输出：以文字描述首页打卡按钮在各种用户场景下的图标状态、允许打卡时间窗口、打卡后状态变化及状态自动失效时机。
- 本文档描述的是**期望（正确）行为**，部分规则与当前代码实现存在差异（已在对应 Scenario 用 `⚠️ 当前实现差异` 标注），后续将据此修改代码。

## Impact
- 受影响代码（仅作为梳理对象，不改动）：[index.vue](file:///d:/wuzuniao/yao/frontend/src/pages/index/index.vue)
- 受影响能力：首页打卡交互、状态机判定、定时刷新机制、打卡记录匹配判定。

## ADDED Requirements

### Requirement: 打卡按钮状态机梳理
系统 SHALL 以文字形式完整描述首页打卡按钮在以下场景的视觉表现与可交互性：未登录、已登录无任务、有任务但不在日期范围、在日期范围但未到打卡时间、可打卡、已完成、刚打卡成功。

#### Scenario: 按钮的四种基础状态
- **GIVEN** 打卡按钮存在四种内部状态：`disabled` / `waiting` / `active` / `done`
- **THEN** 每种状态的视觉表现如下表：

| 内部状态 | 背景色 | 图标资源 | 按钮文字 | 单击是否触发打卡 |
|----------|--------|----------|----------|------------------|
| `disabled` | `#8a8a8a` 灰色 | 不显示图标（文字垂直居中） | "无打卡任务" | 否 |
| `waiting` | `#d97706` 橙色 | **不显示图标**（文字垂直居中） | "未到打卡时间" | 否（但可长按 3 秒重置） |
| `active` | `#d03238` 红色 | `daka_0.png` | "立即打卡" | 是 |
| `done` | `#2ead4b` 绿色 | `daka_1.png` | "已打卡" | 否（含刚打卡成功） |

> 说明：`done` 状态来源有二——①当前提醒时间已存在匹配打卡记录（已完成）；②刚打卡成功后 `justCheckedInTimeId` 与当前 `timeId` 匹配。两者视觉一致。
>
> ⚠️ 当前实现差异：当前代码 `waiting` 状态会显示 `daka_0.png`（因模板 `v-if="!isButtonDisabled"` 仅排除 disabled）。期望为 `waiting` 不显示图标。

#### Scenario: 情况 1 — 用户未登录
- **WHEN** `userStore.userInfo` 为空（`isLoggedIn = false`）
- **THEN** `activePlans` 为空，显示空状态文本"欢迎使用无足鸟按时吃药打卡！"
- **AND** 任务卡片区域隐藏
- **AND** 按钮状态 = `disabled`（灰色"无打卡任务"，不显示图标，不可点击）

---

#### Scenario: 情况 2 — 已登录但无进行中计划
- **WHEN** 已登录，但 `listPlans` 返回的计划中 `status === 1` 的为空（用户未创建计划或全部已停用）
- **THEN** 显示空状态文本"请先到设置界面创建您的打卡计划，常见问题可参考同界面里的帮助中心。"
- **AND** 任务卡片区域隐藏
- **AND** 按钮状态 = `disabled`（灰色"无打卡任务"）

---

#### Scenario: 情况 3 — 有计划但当天不在计划日期范围内
- **WHEN** 已登录、有进行中计划，但今日日期 `today` 满足 `today < plan.start_date` 或 `today > plan.end_date`
- **THEN** 任务卡片显示（主要卡片展示计划名、备注，右上角"进行中"徽章）
- **AND** 按钮状态 = `disabled`（灰色"无打卡任务"）

---

#### Scenario: 情况 4 — 在日期范围内但该计划无任何提醒时间
- **WHEN** 已登录、有计划、在日期范围内，但 `sortedTimes` 为空（`notification_times` 为空数组）
- **THEN** 按钮状态 = `disabled`（灰色"无打卡任务"）

---

#### Scenario: 情况 5 — 在日期范围内但未到任何提醒时间的"开始打卡时间"
- **WHEN** 已登录、有计划、在日期范围内、有提醒时间
- **AND** 当前时间（分钟数 `nowMinutes`）早于所有提醒时间 `t` 的"开始打卡时间" `t.minutes - 60`（即尚未进入任何提醒的可打卡时段）
- **THEN** 按钮状态 = `waiting`（橙色"未到打卡时间"，**不显示图标**，文字垂直居中）
- **AND** 单击不触发打卡，但可长按 3 秒重置为 `active`

> ⚠️ 当前实现差异：当前代码 `waiting` 仅在"不在任何 `[t-60, t]` 窗口内"时触发；期望语义是"未到开始打卡时间 `t-60`"。

---

#### Scenario: 情况 6 — 到达开始打卡时间且当前提醒未打卡（active）
- **WHEN** 当前时间已到达某提醒时间 `t` 的"开始打卡时间" `t.minutes - 60`（即 `nowMinutes >= t.minutes - 60`）
- **AND** 当前提醒时间 `t` **未匹配到打卡记录**（匹配规则见"打卡记录匹配判定"）
- **AND** 当前时间处于该提醒时间的有效打卡区间内（即从 `t-60` 开始，持续到用户打卡为止；即使超过 `t` 整点，只要未打卡仍为 `active`，直至下一个提醒的"开始打卡时间"或当日 24:00）
- **THEN** 按钮状态 = `active`（红色"立即打卡"，显示 `daka_0.png`）
- **AND** 单击可触发打卡

> ⚠️ 当前实现差异：当前代码将 `active` 严格限制在 `[t-60, t]` 窗口内，超过 `t` 即转为 `waiting`。期望为：从 `t-60` 开始持续可打卡，直到用户完成打卡；超过 `t` 仍未打卡仍保持 `active`（未打卡状态），直至下一个提醒开始打卡时间或当日 24:00。

---

#### Scenario: 情况 7 — 进入窗口但当前提醒已打卡（done）
- **WHEN** 当前时间已到达某提醒时间 `t` 的"开始打卡时间" `t.minutes - 60`
- **AND** 当前提醒时间 `t` **已匹配到打卡记录**（匹配规则见"打卡记录匹配判定"）
- **THEN** 按钮状态 = `done`（绿色"已打卡"，显示 `daka_1.png`）
- **AND** 单击不触发打卡，但可长按 3 秒重置为 `active`
- **AND** `done` 状态持续到下列任一条件：
  1. **下一次提醒时间的"开始打卡时间"**：即到达 `next_t.minutes - 60`，进入下一个提醒的可打卡时段，按钮重新判定为 `active`（未打卡）或 `done`（已打卡）；
  2. **当日 24:00 之后**：跨日后重置为 `waiting`（等待次日首个提醒的开始打卡时间）。

> ⚠️ 当前实现差异：当前代码对"小间隔提醒（gap≤120）的前半段"一律判为 `done`，未检查用户是否实际打卡。期望为：前半段若**前一个提醒已打卡**则 `done`，若**前一个提醒未打卡**则保持 `active`（未打卡状态，允许补打卡）。

### Requirement: 打卡记录匹配判定
系统 SHALL 明确说明如何判断"某提醒时间是否已打卡"，用于驱动 `active` / `done` 状态切换。

#### Scenario: 单条提醒时间的匹配规则
- **GIVEN** 计划仅有一条提醒时间
- **WHEN** 判断该提醒是否已打卡
- **THEN** 只要当日存在**任意一条**该计划的打卡记录（不限定具体时间点），即视为该提醒已打卡
- **AND** 状态为 `done`，持续到当日 24:00 后重置为 `waiting`

#### Scenario: 多条提醒时间的匹配规则
- **GIVEN** 计划存在多条提醒时间
- **WHEN** 判断某提醒时间 `t` 是否已打卡
- **THEN** 在 `t` 的**前后各 1 小时**区间 `[t.minutes - 60, t.minutes + 60]` 内存在打卡记录，即视为该提醒时间 `t` 已打卡
- **AND** 多条提醒的打卡记录互不混淆：某条提醒已打卡不影响其他提醒的 `active` 判定

> ⚠️ 当前实现差异：当前代码使用 `checked_time_ids`（按 `plan_time_id` 精确匹配）判定，且仅判断当前 `timeId` 是否在集合中。期望改为按"时间区间匹配"判定，单条提醒看全天任意记录，多条提醒看 `[t-60, t+60]` 区间记录。

### Requirement: 允许打卡时间窗口的明确描述
系统 SHALL 明确说明每个提醒时间的"开始打卡时间"及其有效区间。

#### Scenario: 开始打卡时间定义
- **GIVEN** 提醒时间 `t`（HH:MM），其分钟数为 `m = h*60 + min`
- **THEN** "开始打卡时间" = `m - 60`（提醒时间整点前 60 分钟）
- **AND** 从 `m - 60` 起，按钮进入 `active` 状态（若该提醒未打卡），用户可打卡
- **AND** 可打卡状态**不因超过提醒时间 `m` 而结束**：只要用户尚未打卡，按钮持续保持 `active`（未打卡状态）
- **AND** `active` 状态结束于下列任一条件：
  1. 用户完成打卡 → 转为 `done`；
  2. 到达下一个提醒时间的"开始打卡时间" → 重新判定（针对新提醒）；
  3. 当日 24:00 后 → 重置为 `waiting`。

#### Scenario: 多提醒时间小间隔（gap ≤ 120）的区间划分
- **GIVEN** 相邻两次提醒 `prevT` 与 `t`，间隔 `gap = t.minutes - prevT.minutes ≤ 120`
- **THEN** 中点 `midpoint = (prevT.minutes + t.minutes) / 2`
- **AND** `[t-60, midpoint)` 段：判定"前一个提醒 `prevT` 是否已打卡"
  - 已打卡 → `done`（已完成）；
  - 未打卡 → `active`（保持未打卡状态，允许补打卡）；
- **AND** `[midpoint, ...]` 段：判定"当前提醒 `t` 是否已打卡"
  - 已打卡 → `done`；
  - 未打卡 → `active`。

> 含义：相邻两次提醒间隔较近（≤2 小时）时，前一次提醒的可打卡窗口与下一次提醒的窗口重叠。系统以中点划分归属：前半段服务于前一次提醒（看前一次是否已打卡），后半段服务于当前提醒（看当前是否已打卡）。

### Requirement: 打卡后图标状态变化描述
系统 SHALL 描述用户点击"立即打卡"成功后按钮的状态流转。

#### Scenario: 打卡成功后的即时状态
- **WHEN** 用户在 `active` 状态下点击按钮
- **AND** `createCheckin` 接口返回 `code === 0`
- **THEN** 将该打卡记录纳入"匹配打卡记录"判定
- **AND** `forceActive` 重置为 `false`
- **AND** `justCheckedInTimeId` 设为当前 `timeId`（用于即时反馈，无需等待定时器同步）
- **AND** 按钮立即转为 `done` 视觉（绿色"已打卡" + `daka_1.png`），**不弹窗**提示
- **AND** 单击不再触发打卡（`isCheckinDone` 为 `true` 拦截）

#### Scenario: 打卡成功后 done 状态的持续时长
- **WHEN** 用户完成某提醒时间 `t` 的打卡，按钮转为 `done`
- **THEN** `done` 状态持续到下列任一条件：
  1. **下一次提醒时间的"开始打卡时间"**（`next_t.minutes - 60`）：进入下一个提醒的可打卡时段，按钮重新判定；
  2. **当日 24:00 之后**：跨日重置为 `waiting`，等待次日首个提醒的开始打卡时间；
  3. **切换任务**：`justCheckedInTimeId` 重置为 `null`，按新任务重新判定；
  4. **长按 3 秒重置**：强制转为 `active`（用于补打或纠错）。

### Requirement: 图标状态自动刷新时机描述
系统 SHALL 描述按钮状态随时间刷新的机制与边界。

#### Scenario: 每分钟定时刷新
- **GIVEN** 页面 `onShow` 时启动 `refreshTimer`，间隔 `60000ms`（1 分钟）
- **WHEN** 定时器触发
- **THEN** 调用 `loadTodayCheckins` 从数据库同步当前主要卡片计划的今日打卡记录
- **AND** 根据同步后的"匹配打卡记录"重新判定当前提醒是否已打卡，相应设置 `done` 或 `active`
- **AND** 若数据库连接异常，仅打印日志，不阻塞界面，按钮保持默认状态

#### Scenario: 时间流逝与状态重算的关系（边界说明）
- **GIVEN** `checkinState` 为 `computed`，内部使用 `new Date()` 获取当前时间
- **THEN** `new Date()` 不是 Vue 响应式依赖，**时间流逝本身不会自动触发 `checkinState` 重算**
- **AND** 状态重算由以下事件触发：
  1. 每分钟定时器更新打卡记录（响应式 ref 变化）；
  2. 用户交互（点击次要卡片、选择任务、长按重置、`onShow` 重新加载计划等）引起响应式依赖变化；
  3. `forceActive` / `primaryPlanId` / `justCheckedInTimeId` 等 ref 被修改。
- **NOTE** 因此"跨入新提醒时间窗口"或"到达下一次提醒开始打卡时间"的视觉切换，通常在下一次定时器触发或用户交互时才体现，而非严格在边界时刻即时切换。

#### Scenario: 长按 3 秒重置
- **WHEN** 按钮处于 `waiting` 或 `done` 状态（非 `disabled`、非 `active`）
- **AND** 用户长按按钮 3 秒（`handleLongPress` 启动 3000ms 计时器）
- **THEN** `forceActive` 设为 `true`，强制 `checkinState` 返回 `active`
- **AND** 显示"已重置为立即打卡"成功提示
- **AND** 按钮转为红色"立即打卡"，可点击打卡
- **AND** 打卡成功或切换任务后 `forceActive` 重置为 `false`

#### Scenario: 页面显示时刷新
- **WHEN** 页面触发 `onShow`（包括首次进入及从其他页面返回）
- **THEN** 调用 `loadActivePlans` 加载进行中计划
- **AND** 若存在主要卡片计划，调用 `loadTodayCheckins` 同步今日打卡记录
- **AND** 启动每分钟刷新定时器（若尚未启动）

### Requirement: 任务切换对打卡状态的影响描述
系统 SHALL 描述切换主要卡片任务时打卡状态的联动行为。

#### Scenario: 点击次要卡片互换
- **WHEN** 用户点击次要卡片（`handleSecondaryClick`）
- **THEN** 主要/次要卡片的计划 ID 互换
- **AND** `forceActive` 重置为 `false`
- **AND** `justCheckedInTimeId` 重置为 `null`
- **AND** 重新调用 `loadTodayCheckins` 加载新主要计划的打卡记录
- **AND** 按钮状态根据新计划的提醒时间、当前时间与匹配打卡记录重新判定

#### Scenario: 从任务列表选择任务
- **WHEN** 用户点击"..."展开任务列表并选择某任务（`handleSelectTask`）
- **THEN** 选中任务成为主要卡片，原主要卡片降为次要（若选中任务原为次要则互换）
- **AND** `forceActive` 重置为 `false`，`justCheckedInTimeId` 重置为 `null`
- **AND** 重新调用 `loadTodayCheckins`
- **AND** 按钮状态重新判定
